Feature Creep – Discussion Answers

1. Rank features by usage (which personas actually use them?)

Let’s assume 3 simple personas for your project:
• Ananya (Student, beginner)        → features needed: quick Outfit Suggestions/Recommendations
• Riya   (Working professional)     → features needed: efficient Occasion/Weather-based Outfits
• Kabir  (Fashion enthusiast)       → features needed: wants more control & customization

Feature Ranking (Most → Least Used)
1. Outfit Recommendation System, Preference Selection and Outfit Display Interface → User: ALL personas
2. Filtering Options and Save Outfit Feature                                       → User: Riya & Kabir
3. Weather-Based Suggestions                                                       → User: Ananya & Riya
4. Basic User Profile                                                              → User: Mostly Kabir
5. Advanced customization (if added)                                               → User: Kabir

Potential Feature Creep
• Advanced customization
• Social sharing
• Complex profile system

⸻

2. Where is the main tension in your project?

Main tension: Simplicity vs Functionality
• Some users want:
• “Just tell me what to wear”                  ("ONE CLICK" OPTION = SIMPLE features – Ananya)
• Others want:
• “Let me tweak styles, colors, combinations”  ("CUSTOMIZE ATTIRE" OPTION = Complex features - Kabir)

Concrete Example:
• If the system auto-generates outfits instantly  → simple but less flexible
• If you add too many filters/options             → powerful but confusing

Solution:
• Default setting:    "ONE CLICK" button         - for simple auto-suggestions
• Optional Function:  “CUSTOMIZE ATTIRE” button  - for advanced users

⸻

3. How to avoid feature creep?

Team can follow a simple process:

Step 1: Feature evaluation checklist

  Before adding any feature, ask:
    • Does it solve a common user need?
    • Is it used by at least 2 personas?
    • Is it essential for MVP?

👉 Note: We should have regular "SCRUM" meeting which is coordinated with Stakeholders and sample would be users.


Step 2: Use MVP approach
  • Build only core features first
  • Delay “nice-to-have” / Optional (Additional) features


Step 3: Prioritization method
  • Must-have
  • Should-have
  • Nice-to-have

👉 Only implement Must-have + some Should-have


Step 4: Say “No” with reason while also stating that on further versions there's a possibility of implementation

Example:

“We are not adding social sharing as of now because it does not support the core goal of outfit recommendation.”
